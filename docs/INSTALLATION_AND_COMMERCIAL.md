# Instalacion y Comercial

## Producto local-first

Jarvis Codex debe venderse como una aplicacion local que vive en el computador del cliente. No debe depender de una nube propietaria para pensar, recordar o ejecutar tareas basicas.

## Plataformas

Objetivo inicial:

- Windows: instalador `.exe` o `.msi`.
- macOS: instalador `.dmg` o `.pkg`.

El usuario no debe tener que abrir terminales, levantar servidores ni configurar archivos manualmente.

## BYOK

El modelo comercial inicial debe asumir que el cliente usa su propia llave de modelo. Gemini es el proveedor recomendado por accesibilidad, costo y facilidad de obtencion.

La interfaz no debe decir "trae tu API key" como concepto principal. Debe guiarlo asi:

1. Conectar inteligencia.
2. Abrir guia.
3. Crear llave.
4. Pegar llave.
5. Probar conexion.
6. Activar Jarvis.

## Licencia

La licencia comercial debe resolver:

- Activacion local.
- Limite de dispositivos.
- Marca blanca.
- Transferencia de licencia.
- Revocacion en caso de abuso.
- Modo offline temporal.
- Actualizaciones opcionales.

## Actualizaciones

Aunque Jarvis aprenda y cree capacidades localmente, el producto base igual necesita una estrategia de actualizacion opcional.

Motivos:

- Cambios en APIs de Google.
- Cambios en permisos de macOS o Windows.
- Correcciones de seguridad.
- Compatibilidad con nuevos proveedores.
- Reparacion de conectores rotos.
- Mejoras del motor local.
- Packs verificados de verticales.

Estas actualizaciones no deben ser una dependencia para operar cada dia. Deben ser controladas, visibles y reversibles cuando sea posible.

## Marca blanca

Debe poder configurarse:

- Nombre comercial.
- Logo.
- Voz.
- Tono.
- Rubro.
- Packs iniciales.
- Tutoriales propios.
- Permisos por defecto.
